1. JVM (Java Virtual Machine)
  • It’s run time of Java applications
  • It calls the main() method
  • Part of JRE (Java Runtime Environment)
  • Java is cross-platform because of JVM
  • It creates .class file for each class declared in java files. Each .class file has a corresponding “Class” object, which can be accessed by Java reflections.
  • JVM also contains JNI (Java Native Interface), to call methods from other languages like C or C++
  • What is “native” keyword in Java: Basically declares native method defined in other languages like C/C++: https://stackoverflow.com/questions/6101311/what-is-the-native-keyword-in-java-for
  • JVM Memory:
  1. Method Area: Which stores class level info. Like class name, its parent, methods etc. Its shared resource and 1 per JVM
  2. Heap Area: Shared resource: 1 per JVM, stores Objects
  3. Stack Area: 1 per thread. Every stack frame stores method calls, and local variables to that method call.
  4. PC Registers: 1 per thread. Stores the address of the current execution instruction.
  5. Native Method Stack: Same as stack area, but for native method calls

More than 1 shutdown hook can be registered but order of execution is not guaranteed.

  2. How Java is platform Independent (Portable)?

Java files are compiled in .class files by the compiler. .class files have byte code and not the machine level instructions. This byte code can not be executed directly, it needs an interpreter to execute them. JVM is that interpreter, which can execute byte code.

JVM is platform dependent and available for different OS. So platform dependent JVM and compilation in byte code makes Java platform independent, unlike C/C++, which generates .exe files, which is platform dependent.

  1. Java is statically typed language: Which means that type of all the variables are known at compile time. Once the variable is declared with one type, it can not hold the value of another type. Python, Javascript etc. are dynamically typed language, in which variables can hold any type of data without any declaration.
  2. Type of variables in Java:
  1. Local Variables:
  1. Defined in a block or function or constructor.
  2. Initialized when function is called and destroyed when function call returns.
  3. Scope of those variables is block scope (Can’t called outside of function/block where it’s defined).
  1. Instance Variables:
  1. Non static variables, defined in class outside of any method, constructor or block.
  2. Separate copy per object (each object will have its own copy)
  3. Can have access specifiers (private, public, default, protected)
  4. Changes made in one object do not reflect in other objects copy of these variables as they share separate copy
  1. Static/Class Variables:
  1. defined in class outside of any method, constructor or block with static keyword.
  2. Single copy per class (copy is shared between all the objects).
  3. Can be accessed by directly class name without initializing the object of the class unlike instance variables.
  4. Changes made in one objects do reflects in all the objects as they share a single copy.
  1. Scope of Java variables: Java has block scope unlike javascript. Block refers to for loops block, blocks of “{“ and “}”, try/catch, methods, constructors etc.

Now what is Runtime Polymorphism or Dynamic method dispatch?

  1. Dynamic method is the mechanism by which the call to an overridden method is resolved at runtime instead of compile time.
  2. For code example, refer: https://www.geeksforgeeks.org/dynamic-method-dispatch-runtime-polymorphism-java/
  3. Variables can not be overridden like methods.

Advantage:

This allows a class to specify methods, that will be common to all its derivatives, while allowing its subclasses to define specific implementation of all or some of the methods.

Static vs Dynamic binding:

  1. Static binding is done during compile time, while dynamic binding is done during runtime based on the type of runtime object.
  2. Private, Final and static methods and variables uses static binding, which is done at compile time.
  3. Overloaded methods are resolved (determining which method to call when there are multiple instances of method with the same name) at compile time, while overridden methods are resolved at run time based on the object being referred to.

  1. Association, Composition and Aggregation in Java:
  1. Association is relationship between objects. It can be one to one, one to many, many to one, many to many. It’s generic form of relationship which depicts one object is used by another object.

public class Foo {
   void Baz(Bar bar) {
   }
};

                        This is association: class Foo  uses Bar.

  1. Composition: One class own an object and responsible for its lifetime. When Foo dies, so does Bar. So composition is a “part of” relationship.

public class Foo {
   private Bar bar = new Bar();
}

  1. Aggregation: Class uses an object which is borrowed from someone else (from other class). When Foo dies, Bar may live on. So aggregation is a “has a” relationship.

public class Foo {
   private Bar bar;
   Foo(Bar bar) {
      this.bar = bar;
   }
}

                        

For real life example, consider an institute having multiple departments and each department can have multiple students.

Here each class (Institute, Department, and Student) is associated with each other. Relation between Department and Student is “part of” (Aggregation), which means students are still part of Institute even if department got removed from institute. Relation between Department and Institute is “part of” (Composition). Which means Department can not exist if Institute is destroyed.

Another real life example of Aggregation is Car and its tires. Tires can still exist and can be used even if the car is damaged.

Relationship between body and blood cells is example of Composition. Blood cells can not exist if body is dead.

  1. Object class in Java: It’s root class in class hierarchy. Every class directly or indirectly extends to Object class. Therefore all the method of Object class is available to all the classes. Here are a few methods of object class:
  1. toString(): Used to get String representation of an Object. By default it returns the name of the class and hexadecimal hashcode of the class. Something like this: “String@a12fa0”. One should override this method to get the custom String representations of object. For ex. If there is an object of “Student”, toString() might be overridden to return name and roll number of the student. Here is the default toString() implementation in Object class:

// Default behavior of toString() is to print class name, then
// @, then unsigned hexadecimal representation of the hash code
// of the object
public String toString() {
   return getClass().getName() + "@" + Integer.toHexString(hashCode());
}

                        Whenever we print object reference, it implicitly calls it’s toString() method

  1. hashCode(): For every object, JVM generates a unique number which is called HashCode. This HashCode is not the address of the object. While overriding hashCode method, one should make sure that it generates a unique number for each object. For example: Roll number can be hashCode for student class.
  2. equals(Object o): Compares the given object with “this” reference. Here “this” will refer to the object on which equals() method is called. Override this method to get own equality conditions on objects.
  3. getClass(): Returns the class of the Object. This can’t be overridden as it is final in Object class.
  4. clone(): For cloning the object. Returns exact same copy of the object.
  5. finalize(): This method is called just before the object is garbage collected. Override this method to perform clean-up activities, free up system resources to reduce memory leaks.

  1. Why multiple inheritance is not supported in Java?

The problem with multiple inheritance is that when there is a method with the same signature in both the parent classes, then the compiler cannot determine which method to call or which method gets priority.

The above problem is not diamond problem, as there are two different methods doing different things with the same signature.

Diamond Problem in multiple inheritance: Refer: https://en.wikipedia.org/wiki/Multiple_inheritance#The_diamond_problem for real life example

  A
/ \
B   C
\ /
 D

The diamond problem is an ambiguity that arises when two classes B and C inherit from A, and class D inherits from both B and C. If a method in D calls a method defined in A (and does not override the method), and B and C have overridden that method differently, then from which class does it inherit: B, or C?

The problem with multiple inheritance (diamond problem) is that two classes may define different ways of doing the same thing, and the subclass can't choose which one to pick.
